1. 基本方針の説明（短くまとめる版）

静的解析ツール（例：Polyspace、CodeSonar 等）は、
抽象解釈やデータフロー解析などの形式的手法を用いて
潜在的な不具合や脆弱性を検出する。

しかし、任意のプログラムに対して
すべての脆弱性の有無を完全かつ自動的に判定することは
計算理論上不可能である。

そのため、静的解析結果は
「潜在的欠陥の検出支援」と位置付けるべきであり、
網羅的保証を意味するものではない。

2. 少し踏み込んだ理論的説明（推奨）
■ 理論的制約

任意のプログラムの意味的性質（例：バッファオーバーフローの発生可否）を
完全かつ自動で判定することは、
計算理論（停止問題・Riceの定理）により一般に不可能とされている。

したがって、静的解析ツールは
「見逃しゼロ」かつ「誤検知ゼロ」を同時に満たすことはできない。

■ 実装上のトレードオフ

実用的な解析時間内で動作させるため、
静的解析ツールは以下の近似を行う：

パス数の制限

ループの抽象化

ポインタ解析の簡略化

外部関数のモデル化

値範囲の近似

これらの近似は解析性能と精度のトレードオフであり、
特定条件下では脆弱性が検出されない可能性がある。

② 形式手法の理論的限界
③ 実用ツールにおける近似とトレードオフ
④ その結果として起こり得る検出限界

1. 形式手法の概要

本ツールは、抽象解釈等の形式手法に基づき、
プログラムを実行せずに全実行パスを数学的に近似解析することで、
ランタイムエラーや脆弱性の可能性を検出する。

2. 形式手法の理論的限界

任意のプログラムに対して、
特定の意味的性質（例：バッファオーバーフローの有無）を
完全かつ自動で判定することは、
計算理論上一般に不可能である（停止問題・Riceの定理）。

そのため、静的解析では
見逃し（false negative）または誤検知（false positive）の
いずれかを許容する必要がある。

3. 実用ツールにおける近似

実用的な解析時間内で動作させるため、
以下のような近似が用いられる：

ループの抽象化

ポインタの近似解析

外部関数のモデル化

入力値範囲の保守的推定

これらの近似は解析性能と精度のトレードオフである。

4. 結論（検出限界の説明）

上記の理論的制約および実装上の近似により、
特定条件下では脆弱性が検出されない場合がある。
したがって、本ツールは品質向上の有効な支援手段であるが、
網羅的保証を意味するものではない。

この制約は特定ツール固有の問題ではなく、
静的解析技術一般に共通する理論的制約である。

■ パス解析（Path-sensitive analysis）

プログラムの分岐条件に基づき、
実行可能なパスを探索しながら状態を追跡する解析手法である。

具体的には：

if / switch による分岐

条件ごとの変数値の追跡

エラー状態への到達可能性判定

例：

if (len < 10)
    buf[len] = 0;


このとき CodeSonar は

len < 10 のパス

len >= 10 のパス

を分離して解析します。

■ データフロー解析（Data-flow analysis）

変数の値の伝播（taint、範囲、未初期化など）を
プログラム全体で追跡する解析手法。

具体例：

ユーザ入力 → 変数A → 配列インデックス

NULL値 → ポインタ参照

未初期化値 → 演算

2️⃣ 理論的限界

ここが重要です。

■ パス爆発（Path Explosion）

分岐が増えると：

if (a)
if (b)
if (c)


パス数は指数関数的に増加します。

n個の分岐 → 最大 2ⁿ パス

現実のソフトウェアでは：

ループ

ネスト

条件付きコンパイル

関数呼び出し

により、実行パスは理論的に無限になります。

したがって：

すべてのパスを完全に探索することは計算量的に不可能。

■ 不可決定性

「あるパスでエラーが発生するか？」は
停止問題に帰着します。

つまり：

任意のプログラムについて
エラー到達可能性を完全に判定する一般アルゴリズムは存在しない。

これは静的解析全般の理論的限界です。

3️⃣ 実装上の近似

実用ツールでは以下の制限を設けます。

■ パス制限

探索深さ制限

ループ展開回数制限

複雑な条件の単純化

→ 深い経路は探索されない場合がある

■ 値の近似

例：

len ∈ [0, 100]


と推定された場合、実際には

len = 128


があり得ても、解析がそこまで精密に追えない場合がある。

■ 関数ポインタ・外部関数

動的ディスパッチ

コールバック

外部ライブラリ

はモデル化されるため、実挙動との差異が生じる可能性がある。

4️⃣ 結論としての限界

資料向けまとめ文：

CodeSonar のパス解析およびデータフロー解析は、
実行せずにプログラムの振る舞いを推論する高度な静的解析技術である。

しかし、実行パス数は理論的に指数関数的に増加し、
またエラー到達可能性判定は計算理論上一般に不可決定である。

そのため、実用ツールでは解析深度や値推定に近似を導入しており、
特定条件下では脆弱性が検出されない場合がある。

1️⃣ CodeQLの仕組み

まず仕組みを正しく書きます。

CodeQL は、
プログラムをデータベース化し、
論理クエリによってコードパターンやデータフローを検出する
クエリベースの静的解析手法である。

特徴：

ソースコード → 抽象構文木 → データベース化

クエリ言語（QL）で脆弱性パターンを定義

OSSのためクエリ拡張可能

2️⃣ OSSであることの意味

CodeQL はクエリを利用者が修正・拡張可能であるため、
特定プロジェクト固有の脆弱性パターンに対応した検出ロジックを追加できる。

ここで「他ツールより柔軟」という事実を認めます。

3️⃣ ただし理論限界は変わらない

ここが重要です。

しかし、CodeQL による解析も静的解析の一種であり、
実行パスの爆発的増加や値推定の近似に起因する
計算量的制約を受ける。

つまり：

クエリを修正すれば「パターン」は増やせる

しかし「到達可能性の完全判定」はできない

4️⃣ 具体的に何ができて何ができないか
✔ できること

独自APIに対するチェック追加

特定構造体のサイズ検証ルール追加

プロジェクト固有のラッパ関数をsink/sourceに指定

データフロー追跡範囲の調整

これは確かに強みです。

❌ できないこと

無限ループを含む全パスの完全探索

任意の非線形演算の完全評価

実行時依存値の完全推論

Riceの定理を回避すること

つまり：

クエリを拡張できても、
「静的解析の理論的限界」そのものは回避できない。
